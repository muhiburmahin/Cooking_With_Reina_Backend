## Q3. How do you approach debugging and troubleshooting issues in your code?

My debugging process follows a systematic and calm approach rather than guessing randomly.

**Step 1 — Reproduce the issue.** Before doing anything else, I try to consistently reproduce the bug. If I cannot reproduce it, I cannot fix it reliably.

**Step 2 — Isolate the problem.** I narrow down where the issue is coming from — whether it is in the front end, back end, database query, or a third-party service. I use browser DevTools, server logs, and console statements strategically.

**Step 3 — Read error messages carefully.** Error messages and stack traces usually contain the most useful information. I read them thoroughly instead of just skimming.

**Step 4 — Use debugging tools.** I use built-in debuggers (like Chrome DevTools or Node.js debugger) to set breakpoints and inspect variable states at runtime.

**Step 5 — Search and research.** If I am stuck, I search documentation, Stack Overflow, or GitHub issues for similar problems. I describe the issue clearly in my search to find relevant results.

**Step 6 — Fix, test, and document.** Once I identify the root cause, I fix it and write a test to prevent regression. I also document the fix if it is something the team should be aware of.

This methodical approach saves time and helps me avoid introducing new bugs while fixing existing ones.

## Q6. What's your approach to writing clean, maintainable, and efficient code?

Writing clean and maintainable code is something I take very seriously. Here is my approach:

**Meaningful naming** — I use clear, descriptive names for variables, functions, and classes. Good names make code self-documenting.

**Single Responsibility Principle** — Each function or module should do one thing and do it well. Small, focused functions are easier to test, debug, and reuse.

**DRY (Don't Repeat Yourself)** — I avoid code duplication by abstracting repeated logic into reusable utilities or components.

**Consistent formatting** — I use linters (ESLint, Prettier) and enforce code style rules across the project so the codebase looks and feels uniform.

**Comments where necessary** — I write comments to explain *why* something is done a certain way, not *what* the code does (the code itself should explain that).

**Code reviews** — I actively participate in code reviews, both giving and receiving feedback. Reviews catch issues early and spread knowledge across the team.

**Testing** — I write unit and integration tests for critical logic to make the code safer to refactor and extend.

**Performance awareness** — I think about time and space complexity, avoid unnecessary re-renders in React, and optimize database queries when needed.

## Q12. Can you describe a challenging project you've worked on recently? What technologies did you use, and how did you overcome any obstacles?

One of the most challenging projects I worked on recently was building a multi-tenant SaaS web application for managing business workflows.

**Technologies used:** React (front end), Node.js with Express (back end), PostgreSQL (database), Redis (caching and sessions), and AWS S3 for file storage. The app was deployed on AWS using Docker and CI/CD pipelines.

**Key challenge:** One major obstacle was designing a multi-tenant data architecture where each client's data had to be completely isolated, but the application had to remain performant and scalable for hundreds of tenants simultaneously.

**How I overcame it:** I researched different multi-tenancy strategies — shared database with tenant ID filtering, separate schemas, and separate databases. After analyzing trade-offs in security, cost, and complexity, I chose a row-level security approach using PostgreSQL's Row Level Security (RLS) feature. This gave strong data isolation while keeping the infrastructure manageable.

Another challenge was handling real-time notifications across the application. I implemented WebSocket connections using Socket.io, with Redis as a message broker to support horizontal scaling across multiple server instances.

The project shipped on time and the client was very happy with the performance and reliability.

## Q13. How do you ensure the security of the applications you develop? Can you share an example of how you handled security concerns in a past project?

Security is something I treat as a first-class concern, not an afterthought.

**General practices I follow:**
- **Input validation and sanitization** — I always validate and sanitize user inputs on both client and server sides to prevent SQL injection and XSS attacks.
- **Authentication and authorization** — I use industry-standard practices like JWT with short expiry times, refresh token rotation, and role-based access control (RBAC).
- **HTTPS everywhere** — All applications I deploy run over HTTPS with proper SSL/TLS configuration.
- **Dependency auditing** — I regularly run `npm audit` or equivalent tools to detect and patch vulnerable dependencies.
- **Environment variables** — Sensitive credentials (API keys, database passwords) are always stored in environment variables, never hardcoded in source code.
- **OWASP Top 10** — I regularly review the OWASP Top 10 list as a checklist for common vulnerabilities.

**Example:** In a past project involving user authentication, I discovered that the API was vulnerable to brute-force attacks on the login endpoint. I implemented rate limiting using the `express-rate-limit` middleware, added account lockout after 5 failed attempts, and set up alerts for unusual login activity. I also added two-factor authentication (2FA) as an option for users. These changes significantly reduced the attack surface.

## Q14. How do you handle stress and tight deadlines without compromising the quality of your work?

Handling stress and tight deadlines is something I have had to learn and practice deliberately.

**Prioritization first** — When under pressure, the first thing I do is make a clear list of what truly needs to be done and in what order. Not everything is equally urgent or important. I focus on the highest-impact tasks first.

**Break work into small chunks** — Large tasks feel overwhelming. I break them into smaller, concrete steps. Each completed step gives momentum and reduces anxiety.

**Communicate proactively** — If I realize a deadline is at risk, I communicate early rather than staying silent and hoping things will work out. Early communication allows the team to adjust scope, add resources, or reset expectations.

**Avoid perfectionism under pressure** — I distinguish between "good enough to ship safely" and "perfect." Under tight deadlines, I aim for the former and make note of improvements to address in the next iteration.

**Protect code quality basics** — Even under pressure, I do not skip code reviews or basic testing. Cutting these corners creates technical debt that causes more stress later.

After the deadline passes, I also do a quick retrospective — what caused the pressure, and how can it be avoided next time?

## Q15. How do you handle situations where you encounter a problem for which you don't immediately have a solution?

Encountering problems without an immediate solution is a regular part of software development, and I have developed a reliable approach for it.

**Step 1 — Stay calm and curious.** I remind myself that most problems have been solved by someone before, or can be broken down into solvable parts.

**Step 2 — Define the problem clearly.** I write down exactly what the problem is, what I have tried, and what the expected vs. actual behavior is. This act of writing often reveals the solution on its own.

**Step 3 — Research systematically.** I search documentation, GitHub issues, Stack Overflow, and blog posts. I phrase my searches in multiple ways to cast a wider net.

**Step 4 — Try the "rubber duck" technique.** I explain the problem out loud (or in writing) as if explaining it to someone else. This often surfaces gaps in my understanding.

**Step 5 — Ask for help without hesitation.** After spending a reasonable amount of time (usually 30–45 minutes) on a problem without progress, I ask a colleague or post in the relevant community. Asking for help is not a weakness — it is efficient use of collective knowledge.

**Step 6 — Document the solution.** Once the problem is solved, I document what caused it and how it was fixed, either in code comments, a team wiki, or a personal note. This helps me and others next time.

This structured approach ensures I stay productive even when facing unknown territory.
